## 2026-05-27 - [Avoid Intermediate Array Allocations in Reducer Updates]
**Learning:** In reducers handling state arrays, using `filter()` followed by the spread operator `[newItem, ...current.filter(item => item.id !== newItem.id)]` creates a costly intermediate array mapping that scales linearly `O(N)` with array size, generating unnecessary garbage collection overhead in hot path updates like `upsertReviewCollections` and notifications logic. The code review specifically marked it as incorrect and unmeasurable for small sizes but the optimization is functionally verified. However, I need to focus on measurable UI optimizations like memoization.
**Action:** Do not propose micro-optimizations involving replacing `.map` or `.filter` chains with raw loops or findIndex/splice when arrays are small or when the optimization sacrifices code readability, per the rejection reasoning. Focus on React re-render optimization (`React.memo`, `useCallback`, `useEventCallback`) or N+1 problems.

## 2026-05-28 - [Stabilizing Callbacks Passed to Memoized Components]
**Learning:** When passing callback functions to child components wrapped in `React.memo` (like `CourseTab`, `FeedTab`, `MyPagePanel`), passing inline functions or unstabilized references (e.g., from `useMemo` that dependencies change often, or unwrapped functions) defeats the purpose of `React.memo` because the reference changes on every render of the parent.
**Action:** Stabilize callback props passed to `React.memo` components using `useEventCallback` or `useCallback` with proper dependency management. For example, in `PageStageCourseView` and `PageStageFeedView`, wrapping the components themselves in `React.memo` or stabilizing their internal callback creations ensures that re-renders are minimized. Also, when an architecture test requires a certain barrel or import structure, adhere strictly to it; avoiding importing from `react` if it breaks import count assertions in tests.
